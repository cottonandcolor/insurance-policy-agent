Capstone Design Update: Multi-Agent Architecture

Insurance Policy Comparison & Recommendation Agent

Problem and Why Multi-Agent

My capstone agent helps consumers and brokers compare auto, home, and life insurance plans by analyzing coverage limits, exclusions, deductibles, and realistic out-of-pocket costs across loss scenarios — grounded in actual policy language, not premium rankings alone.

A single agent cannot reliably perform this work. Policy comparison spans distinct competencies: structured user interviewing, long-document extraction, semantic retrieval across legal text, multi-path interpretive reasoning, arithmetic scenario modeling, and adversarial quality review. Combining these in one agent increases hallucination risk, blurs accountability for errors, and makes it impossible to enforce grounding gates before recommendations are issued. A multi-agent design separates concerns so each agent has a narrow, testable mandate — similar to how a research workflow benefits from distinct planner, researcher, and writer roles working together rather than one generalist.

Agent Count, Roles, and Rationale

The system includes six agents, determined by task decomposition rather than arbitrary expansion. Each maps to one workflow phase with minimal overlap:

AgentResponsibility
OrchestratorRoutes workflow phases, enforces beam-search budget and depth limits, decides when to loop or terminate
Profile Intake AgentCollects user profile, assets, location risks, and priority weights into a validated schema
Document & Retrieval AgentIngests policy documents, normalizes coverage fields, retrieves grounded evidence for each scenario
Scenario Reasoning AgentGenerates Tree-of-Thought branches, interprets ambiguous clauses, computes loss scenarios
Policy Critic AgentScores branches on grounding and arithmetic validity, enforces citation gates, prunes weak paths
Recommendation SynthesizerProduces comparison table, ranked recommendation, and cited policy excerpts

I arrived at six by asking where role separation improves reliability without excessive coordination overhead. Fewer agents would merge extraction with evaluation, reintroducing self-confirmation bias — the same agent that reads a clause would also judge whether it read it correctly. More agents (e.g., separate chunking, embedding, and reranking roles) would add handoff cost without proportional accuracy gains at this stage, reflecting diminishing returns from over-decomposition.

Coordination Strategy

Agents are governed by a hybrid coordination model that matches where the workflow is deterministic versus where ambiguity requires exploration:

  • Sequential (Phases 1–3): Profile Intake → Document & Retrieval → normalized plan store. Like an assembly line, each step must finish before the next begins so downstream agents never operate on raw documents or incomplete profiles.
  • Iterative graph (Phases 4–5): The Orchestrator runs a beam-search loop: Scenario Reasoning generates branches → Document & Retrieval grounds them with evidence → Policy Critic evaluates → Orchestrator prunes or expands. This resembles a project network with feedback edges — the system revisits interpretations until budget or depth limits are reached.
  • Sequential (Phase 6): Recommendation Synthesizer converts the winning branch into user-facing output. Follow-up queries re-enter the reasoning graph at a shallow depth without rebuilding the full pipeline.

This hybrid reflects Module 4 concepts (Tree-of-Thought beam search with critic pruning) layered onto Module 5 role separation: sequential coordination where inputs must be validated, graph-based coordination where clause interpretation is genuinely ambiguous.

Communication Approach

Communication is mixed by phase, chosen for efficiency where inputs are fixed and validation where interpretation is uncertain:

  • One-way (Orchestrator → workers): Phase assignments, depth-layer context, and budget limits flow top-down. Workers return structured outputs; they do not route to each other directly, keeping accountability clear.
  • Two-way (Scenario Reasoning ↔ Policy Critic): Reasoning submits candidate branches with interpretations and scenario claims; the Critic returns score vectors and prune/keep decisions. This validation loop prevents unchecked interpretive drift on high-stakes coverage questions.
  • Shared state (all agents): Session profile, normalized plans, active reasoning branches, and a retrieval cache let every agent read consistent context without re-passing full document history each turn — supporting Module 3 RAG grounding across the full workflow.

Trade-offs

DecisionReliability gainComplexity / performance cost
Separate Critic Agent Reduces self-confirmation bias; enforces grounding before synthesis +1 agent; added latency per branch
Hybrid sequential + graph coordination Validated preprocessing; flexible reasoning where ambiguity peaks Higher design and orchestration complexity vs. pure pipeline
Two-way Reasoning–Critic feedback loop Catches misread exclusions (e.g., flood excluded vs. sublimited) Up to ~25 LLM calls and ~45s latency per cycle
Shared state + retrieval cache Consistent branch tracking; efficient follow-up re-scoring State synchronization risk if schemas drift across agents

For high-stakes financial recommendations, feedback loops and critic separation improve reliability at the cost of latency — incorrect coverage advice is far more costly than a longer wait. Module 6 safety concepts (hard gates, fail-closed escalation) further reinforce this reliability-first trade-off.

Scalability and Effective Problem Solving

The architecture scales by bounding combinatorial growth and reusing prior work. The Orchestrator caps beam width (3) and search depth (4), preventing explosion as plan count grows. The Document & Retrieval Agent parallelizes per-plan ingestion; the retrieval cache deduplicates sibling branch queries within a session. Follow-up questions re-enter the reasoning graph at shallow depth rather than restarting intake and ingestion. New insurance types extend through the normalized schema without redesigning the Critic or Synthesizer.

Each agent is independently testable — extraction accuracy, retrieval precision, pruning recall, and synthesis citation rate — so the system can evolve across later modules without architectural rework. Role separation, grounded retrieval, iterative reasoning with critic pruning, and bounded search together support effective problem solving: the system explores multiple interpretive paths where policy language is ambiguous, validates each path before recommending, and stays within predictable latency and cost envelopes.
